## 三、P1 问题（应该修）

### P1-001：点击历史情绪卡只切换 UI，不同步切换音乐

- **文件**：`frontend/src/App.tsx` 第 226-227 行；`frontend/src/components/HistoryPanel.tsx` 第 35 行
- **问题**：历史卡片点击后直接执行 `setCurrentCard`，页面右侧情绪卡会变成历史记录，但当前播放中的音乐不会切换到该历史卡对应的音乐参数。
- **风险**：用户看到的是 A 情绪卡，听到的可能仍是 B 情绪音乐，造成“卡片和音乐不一致”的体验问题。
- **修复建议**：新增 `handleSelectHistory(card)`，内部先 `stopSoundscape()`，再 `setCurrentCard(card)`，并按当前产品逻辑自动调用 `createSoundscape(card.music)`，同步更新 `isPlaying` 和提示语。

### P1-002：连续循环播放会长期累积 oscillator 和 timer 引用

- **文件**：`frontend/src/services/soundscapeEngine.ts` 第 56、70-76、116-149、159-166 行
- **问题**：每个音符都会 `oscillators.push(osc)`，每个定时器都会 `timers.push(timer)`；62 秒循环后继续追加，新一轮不会清理上一轮已经结束的 oscillator 和已触发 timer。
- **风险**：页面长时间播放同一种心情音乐时，引用数组持续增长，停止播放时也需要遍历大量已结束节点。短时间演示影响不明显，长时间使用会带来内存和性能压力。
- **修复建议**：给短音符设置 `osc.onended` 后从数组移除；定时器触发后从 `timers` 移除；或把每轮循环的节点单独归组，在进入下一轮前清理上一轮已结束引用。

### P1-003：自动生成流程缺少“最后一次输入 wins”的防抖保护

- **文件**：`frontend/src/App.tsx` 第 70-104、138-153 行
- **问题**：`handleGenerate(true)` 内部还有 360ms `setTimeout`，外层自动触发 effect 也有 450/650ms 延迟；面部信号持续刷新时，旧的闭包可能在新信号之后才完成，导致重复停止/启动音频或旧结果覆盖新结果。
- **风险**：摄像头识别频繁更新时，情绪卡和音乐可能短时间跳变，尤其在用户反馈“页面跳动”这一类场景下会加重不稳定感。
- **修复建议**：用 `generationIdRef` 或 `AbortController` 模式标记当前生成任务，只允许最后一次任务落盘和播放；同时在 `generating` 时合并相邻触发，而不是每次都 stop/play。

## 四、P2 问题（锦上添花）

### P2-001：依赖版本使用 `latest`，复现稳定性不足

- **文件**：`frontend/package.json` 第 13-20 行
- **问题**：React、Vite、TypeScript、lucide-react 都使用 `latest`。
- **风险**：虽然当前 `package-lock.json` 能锁住本机安装结果，但后续手动重装或更新 lockfile 时容易引入不可预期的破坏性升级。
- **修复建议**：把依赖改成明确版本范围，例如 `^19.x`、`^8.x`、`^5.x`，并在 README 中说明 Node 版本。

### P2-002：摄像头/语音不可用时的提示仍提到“文本输入”

- **文件**：`frontend/src/components/FaceCapture.tsx` 第 55-75 行；`frontend/src/components/EmotionInput.tsx` 第 25-30 行
- **问题**：当前输入框按用户要求改成只读，但摄像头错误提示仍写“可在文本框输入心情描述”。
- **风险**：能力不可用时用户会看到不可执行的恢复路径。
- **修复建议**：改成“请检查浏览器权限，或刷新后重新开启面部识别和声音对话”；如果需要兜底，应设计一个明确的只读外部导入或测试模式，而不是提示手输。

### P2-003：缺少自动化回归测试

- **文件**：`frontend/src/services/*.ts`、`frontend/src/components/*.tsx`
- **问题**：目前只有构建检查，没有单元测试或组件测试。
- **风险**：情绪分析、历史去重、音乐推荐、音频状态切换这些核心逻辑后续修改时容易回归。
- **修复建议**：优先给 `analyzeEmotion`、`recommendMusic`、`saveCard/listCards` 加 Vitest 单元测试；再补一个 Playwright 冒烟测试覆盖首页渲染和点击“开启面部识别”的降级路径。

## 五、架构偏差（与 SDD 不符）

这些不是代码缺陷，主要来自用户后续需求变更，建议在进入 /retro 前同步回写 PRD/SDD。

### AD-001：输入方式从“手动文本/手动情绪 + 语音”改为“语音识别只读输入”

- **SDD 定义**：`EmotionInput` 包含 `onTextChange`、`onManualEmotion`、`onStartSpeech`。
- **实际实现**：输入框 `readOnly`，不再提供手动情绪选择和声音按钮，由点击面部识别自动启动语音。
- **建议**：更新 SDD 的 M-005、全局业务流程和边界条件。

### AD-002：音频从 10-20 秒短音景改为持续循环播放

- **SDD 定义**：`createSoundscape` 按 10-20 秒规划音符，返回短音景。
- **实际实现**：每轮 62 秒，并在同一情绪的 5 首虚拟曲目间循环。
- **建议**：把 M-009 改为“连续情绪音乐引擎”，补充循环、释放和音量策略。

### AD-003：5 首曲名不再展示，但仍用于循环逻辑

- **SDD 定义**：结果卡展示 3 个虚拟曲目。
- **实际实现**：`EmotionCard` 隐藏曲目名称，`MusicProfile.tracks` 仅被音频循环使用。
- **建议**：在 SDD M-011 中明确“曲目列表不展示，仅作为内部循环素材”。

### AD-004：面部识别从页面卡片改为固定悬浮窗口

- **SDD 定义**：`FaceCapture` 是普通页面组件。
- **实际实现**：`.floating-camera-panel` 固定在右上角/移动端右下角，类似视频通话画中画。
- **建议**：补充悬浮层布局、遮挡规则和移动端适配规则。

### AD-005：新增宠物阿狸互动模块

- **SDD 定义**：未定义宠物模块。
- **实际实现**：`PetAli` 固定在页面左下角，根据情绪变化展示动画和文案。
- **建议**：新增模块 M-016，定义输入、状态、交互和响应式布局边界。
